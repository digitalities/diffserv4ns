# NS2_PATCHES — DiffServ4NS modifications to stock ns-2.29

> **Plan completeness check:** The authoritative file list for DS4 patches is
> the `PATCHED_FILES` array in `scripts/patch-ns2-diffserv.sh` (14 entries).
> Any port plan must cover these 14 + `src/diffserv/*` as layer 1.
> Do not infer the DS4 patch set from memory or summary — derive it from
> diffing `src/ns-2.29/` against the pristine target tree.

This document catalogues every modification that DiffServ4NS made to
stock ns-2.29 source files.  The `src/diffserv/` directory is entirely
new code (not patched); this document covers only the changes to
existing ns-2 files.

Generated by diffing `ns2/ns-2.29-pristine/` (ns-allinone-2.29.3)
against `src/ns-2.29/`.

## Detailed patches

### 1. Packet metadata: `common/packet.h`

**What changed:** Two new fields and their accessors were added to
`hdr_cmn` (the common packet header struct).

```diff
+        double  sendtime_;
+        packet_t app_type_;
```

Accessors (lines 518–519 in the modified file):

```diff
+        inline double& sendtime() { return (sendtime_); }
+        inline packet_t& app_type() { return (app_type_); }
```

**Why:** Every ns-2 DiffServ packet needs two pieces of metadata that
stock ns-2 does not carry:

1. `sendtime_` — the wall-clock time the packet was created by the
   sending application, used by `LossMonitor` to compute one-way delay
   (OWD) and IP delay variation (IPDV).  This is *not* the same as
   `ts_` (the scheduler timestamp) or the RTP timestamp.

2. `app_type_` — the application-layer packet type (e.g. `PT_TELNET`,
   `PT_HTTP`, `PT_CBR`), used by the DiffServ policy module
   (`dsPolicy.cc`) to classify packets into per-hop behaviours.  Stock
   ns-2 only carries the transport-layer `ptype_` (e.g. `PT_TCP`,
   `PT_UDP`), which is insufficient for DiffServ classification.

**ns-3 strategy:** Both fields become lightweight ns-3 `Tag` objects
(`DiffServSendTimeTag`, `DiffServAppTypeTag`) attached per-packet.  See
ADR-0005.

### 3. TCP state visibility: `tcp/tcp.h`, `tcp/tcp.cc`

**tcp.h — 2 new fields in `hdr_tcp`:**

```diff
+        int cwnd_;              /* current window - sent for statistic purpose SA*/
+        double t_rtt_;          /* current RTT - sent for statistic purpose SA*/
```

Accessors:
```diff
+        int& cwnd() { return (cwnd_); }
+        double& t_rtt() { return (t_rtt_); }
```

**tcp.cc — 1 hunk in `output()` (line 675):**

```diff
+        tcph->cwnd() = cwnd_;
+        tcph->t_rtt() = t_rtt_;
```

**Why:** The monitoring layer (thesis Chapter 3.3.4) records TCP
congestion-window and RTT alongside loss and delay statistics.  Stock
ns-2 does not expose these in the packet header — they stay inside the
TCP agent.  DiffServ4NS copies them into each outgoing TCP segment so
that `dsMonitor.cc` (and downstream scripts) can correlate delay/loss
with TCP state.

**Note:** `cwnd_` in the TCP header is declared `int` but the TCP
agent's `cwnd_` is a `double`.  The truncation was intentional (the
monitoring only needs integer-window granularity).

**ns-3 strategy:** `DiffServTcpCwndTag` (uint32_t) and
`DiffServTcpRttTag` (Time).  These are only needed for Phase 6
monitoring — they can be deferred.

### 4. Sendtime injection: `apps/udp.cc`

**2 hunks (lines 103, 119 in modified file):**

Both `sendmsg()` code paths (with and without data payload) add:
```diff
+                hdr_cmn::access(p)->sendtime_ = local_time;
```

**Why:** UDP packets need their creation timestamp stamped at the
application layer, immediately after `allocpkt()`.  The `sendtime_`
field in `packet.h` is the vehicle; the UDP agent writes `local_time`
(the Scheduler clock at send time) into it.

**ns-3 strategy:** The DiffServ helper for UDP applications will attach
`DiffServSendTimeTag` with `Simulator::Now()`.

These four files follow the same pattern: call `set_pkttype()` and/or
`set_apptype()` to tag packets with their application identity.

**telnet.cc — `start()` (line 67):**
```diff
+        agent_->set_pkttype(PT_TCP);
+        agent_->set_apptype(PT_TELNET);
```
Sets the transport type to TCP and the app type to TELNET.

**cbr_traffic.cc — `init()` (lines 93–96):**
```diff
-		    agent_->get_pkttype() != PT_TFRC)
-			agent_->set_pkttype(PT_CBR);
+		    agent_->get_pkttype() != PT_TFRC) {
+			agent_->set_pkttype(PT_CBR);
+                         agent_->set_pkttype(PT_UDP);
```

**Bug note:** This sets `pkttype` to `PT_CBR` and then immediately
overwrites it with `PT_UDP`.  The `PT_CBR` line is dead code.  The
intent was probably `set_apptype(PT_CBR)` on the second line.  The
ns-2 simulations still worked because `dsPolicy.cc` classifies on
`app_type_` (set elsewhere) rather than `ptype_`.  This bug does not
need to be reproduced in the ns-3 port.

**realaudio.cc — `init()` (lines 129–130):**
```diff
-	if (agent_)
+	if (agent_) {
 		agent_->set_pkttype(PT_REALAUDIO);
+                agent_->set_pkttype(PT_UDP);}
```
Same pattern as CBR — overwrites with `PT_UDP`.  The app_type is set
elsewhere (via the two-argument `Agent` constructor or Tcl `set_apptype`).

**webtraf.cc — 2 hunks (lines 307, 325):**
```diff
+        a->set_apptype(PT_HTTP);
```
Both `create_source_agent()` and `create_sink_agent()` tag web traffic
agents with `PT_HTTP` so the classifier can identify HTTP flows.

**ns-source.tcl — FTP (line 51):**
```diff
+        [$self agent] set_apptype 27
```
Sets FTP app type to 27 (the numeric value of `PT_HTTP` in ns-2's
packet type enum — likely a mis-label; FTP and HTTP share the type
because the thesis treats them both as "web" traffic for classification).

**ns-3 strategy:** Each ns-3 application helper calls
`packet->AddTag(DiffServAppTypeTag(...))` during packet creation.
The numeric enum is replaced by a string or TypeId-based identifier.

### 6. Monitoring: `tools/loss-monitor.h`, `tools/loss-monitor.cc`

This is the largest set of changes (~150 lines added) and constitutes a
mini monitoring framework grafted onto the existing `LossMonitor` agent.

**loss-monitor.h — new class and new members:**

1. New `tFD` class (frequency distribution histogram):
```cpp
class tFD {
public:
    tFD(double RL, char* FN);
    void occurency(double Value);
    void flush(double norm, double threshold);
private:
    vector<int> FDtable;
    double RangeLenght;     // sic: original spelling
    char FileName[20];
    int counter;
};
```

2. New `LossMonitor` members:
```diff
+       int flowid_;            // flow ID to monitor
+       int npkts_flowid_;      // packet count for per-flow stats
+       double sum_ipdv_;       // running IPDV sum
+       double sum_owd_;        // running OWD sum
+       double current_owd_;    // most recent OWD sample
+       double current_ipdv_;   // most recent IPDV sample
+       double min_owd_;        // minimum OWD observed
+       double previous_owd_;   // previous OWD (for IPDV calc)
+       tFD *owdFD, *ipdvFD;   // frequency distribution objects
```

**loss-monitor.cc — 3 major additions:**

1. Constructor initialisation of all new members + Tcl `bind()` calls
   for `flowid_`, `npktsFlowid_`, `sumIpdv_`, `sumOwd_`, `ipdv_`,
   `owd_`, `minOwd_`.

2. Per-packet OWD/IPDV computation in `recv()`:
   - Filters by `flowid_`
   - Computes `current_owd_ = now - sendtime_` (one-way delay)
   - Tracks `min_owd_`
   - Computes `current_ipdv_ = |current_owd_ - previous_owd_|`
     (instantaneous IPDV per RFC 3393)
   - Accumulates `sum_owd_` and `sum_ipdv_` for averaging
   - Feeds samples into frequency distribution histograms

3. New Tcl commands:
   - `flushFD` (2, 4, and 6-arg variants) — write histograms to files
   - `FrequencyDistribution range_owd range_ipdv suffix` — create
     histogram objects

4. `tFD` implementation — simple histogram with configurable bucket
   width, file output, and percentage-based threshold filtering.

**Why:** The thesis (Chapter 3.3.4) describes a monitoring framework
that measures per-flow OWD, IPDV, and their distributions.  This is
the implementation of that framework, built on top of the existing
`LossMonitor` agent rather than as a separate class.

**ns-3 strategy:** The porting approach for the monitoring subsystem
depends on ADR-0008 (v1 monitoring scope):

- *Option (a) — faithful port:* OWD/IPDV computation becomes a new
  ns-3 helper class reading `DiffServSendTimeTag`; `tFD` becomes a
  `FrequencyDistribution` helper; `DiffServTcpCwndTag`/`TcpRttTag`
  are needed.
- *Option (b) — ns-3 native:* Replace with `FlowMonitor` + trace
  sources on the DiffServ queue disc; cwnd/rtt available via
  `TcpSocketBase` trace sources; no additional Tags needed.
- *Option (c) — defer:* Not ported in v1.

See ADR-0008 for the full trade-off analysis.  This does not block
Phases 1–3 (metering, classification, edge/core routers).

### 8a. Upstream warning-hygiene fixes preserved in ns-2.35 adaptations

**Not DiffServ4NS additions — retained Nortel artefacts cleaned up by upstream ns-2.35.**

Two DS4 diffserv files carried cosmetic issues (dead variables, a stray semicolon)
that upstream ns-2.35 silently fixed but that DS4's base layer would re-introduce
when patched on top. These are preserved via `src-ns235/diffserv/` adaptations that
replace the DS4 base files for ns-2.35 builds. See `docs/NS2_UPSTREAM_DRIFT.md`.

**`dsred.cc::deque()` — dead variables:**  
DS4's rewritten `dsREDQueue::deque()` retained from the Nortel original two local
variables that were already dead in ns-2.29: `hdr_ip* iph` and `int fid`
(`fid = iph->flowid()/32` was never read after assignment). Upstream ns-2.35
removed them. The `src-ns235/diffserv/dsred.cc` adaptation applies the same
four-line removal to suppress `-Wunused-variable` warnings under GCC 4.x.

**`dsEdge.cc` — dead variable and stray semicolon:**  
`edgeQueue::enque()` declared `int codePt` and assigned `codePt = policy.mark(pkt)`
but never read `codePt`; the return value is discarded by calling `policy.mark(pkt)`
directly. Additionally, `edgeQueue::command()` ended with `};` (stray semicolon on
function close). Both are fixed in `src-ns235/diffserv/dsEdge.cc`. No behavioural
impact; purely compiler-warning suppression.

## ns-3 porting implications

### Tag candidates (ADR-0005)

The following per-packet metadata fields were added to ns-2 headers
and need ns-3 Tag equivalents:

| # | Field | ns-2 type | ns-3 Tag | Source | Used by | Status |
|---|---|---|---|---|---|---|
| 1 | `sendtime_` | `double` | `DiffServSendTimeTag` | `packet.h` (hdr_cmn) | `loss-monitor.cc` for OWD | **Accepted** (ADR-0005) |
| 2 | `app_type_` | `packet_t` | `DiffServAppTypeTag` | `packet.h` (hdr_cmn) | `dsPolicy.cc` for classification | **Accepted** (ADR-0005) |
| 3 | `cwnd_` | `int` | `DiffServTcpCwndTag` | `tcp.h` (hdr_tcp) | `dsMonitor.cc` for TCP stats | **Conditional** on ADR-0008 |
| 4 | `t_rtt_` | `double` | `DiffServTcpRttTag` | `tcp.h` (hdr_tcp) | `dsMonitor.cc` for TCP stats | **Conditional** on ADR-0008 |

Tags 1–2 are needed by Phase 2 (classification) and Phase 3 (edge router).
Tags 3–4 are only needed if ADR-0008 resolves to option (a) — faithful
monitoring port.  If ns-3 native monitoring is used (option b), TCP cwnd
and RTT are available via `TcpSocketBase` trace sources without Tags.

### Renames and removals between Nortel original and DiffServ4NS

The `dsPolicy.{h,cc}` files in DiffServ4NS differ from the Nortel
original not only by addition (new features documented above) but also
by renaming and removing some Nortel-original policy classes.  These
changes were undocumented until 2026 and were surfaced during Phase 6
preparation by comparing `ns2/ns-2.29-pristine/diffserv/dsPolicy.{h,cc}`
against `src/ns-2.29/diffserv/dsPolicy.{h,cc}`.

| ns-2.29-pristine (2005/2006) | DiffServ4NS | Direction of change |
|---|---|---|
| `SFDPolicy` (`#define SFD 6`) | `FWPolicy` (`#define FW 6`) | **Sergio renamed SFD → FW** in 2001. Algorithm byte-for-byte identical. Slot 6 preserved. |
| `EWPolicy` (`#define EW 7`) | *(absent)* | **Not merged.** Early Warning policy was added upstream after 2001 and never pulled into the DiffServ4NS fork. |
| `DEWPPolicy` (`#define DEWP 8`) | *(absent)* | **Not merged.** Same as above. |
| `flow_entry.src_id`, `.dst_id` | *(removed)* | **Sergio simplified.** SFD identified flows by `fid`+`src`+`dst`; FW uses `fid` only. |
| `nullPolicy` / `NullPolicy` | `dumbPolicy` / `DumbPolicy` (`#define DUMB 0`) | **DiffServ4NS kept the older name.** Upstream ns-2 renamed `dumbPolicy → nullPolicy` on 27 Nov 2003 (per ns `CHANGES.html`, suggested by Alexander Sayenko); DiffServ4NS never merged this change. |

The `SFDPolicy` → `FWPolicy` rename is the most significant.  "SFD"
stands for "Short Flow Differentiating" (Chen & Heidemann, 2003).
"FW" was introduced in DiffServ4NS (2001) but its expansion is not
recorded in the source code, the thesis, or the ns-users mailing list
archive; the author does not recall the intended meaning.  The
underlying code — including the three penalty modes selected by
`downgrade2` (0=deterministic, 1=probabilistic, 2=periodic) — is
unchanged from the Nortel original.

#### Branch divergence (added 2026-04-16)

The `dumb/null` and `EW/DEWP` rows above establish that DiffServ4NS is a
**divergent 2001 fork** of the `ns-2.1b8a` DiffServ module.  Sergio developed
DiffServ4NS in 2001 against `ns-2.1b8a`, and re-released it on SourceForge
in 2006 ported to ns-2.29, but never re-merged upstream changes to
`diffserv/*` in between.  Evidence:

- If DiffServ4NS had pulled from mainline after 27 Nov 2003, the class
  would now be `nullPolicy`, not `dumbPolicy`.  It is still `dumbPolicy`,
  and the Tcl configuration token is still `"Dumb"`.
- If DiffServ4NS had pulled from mainline, `EWPolicer` and `DEWPPolicer`
  would be in the `policerType` enum (they are in ns-2.29-pristine).
  They are absent.
- Conversely, mainline ns-2 never renamed `SFDPolicy → FWPolicy`
  (ns-2.29-pristine still has `SFDPolicy`).  So the rename is provably
  Sergio's, not an upstream event.

This matters for any attempt to reason about what DiffServ4NS "inherits"
from ns-2 mainline: the answer is *the ns-2.1b8a Nortel/Chen snapshot only*.
Later upstream features and bug fixes that touch `diffserv/*` did not
reach DiffServ4NS users.  Users who rebuild DiffServ4NS against ns-2.29
get a hybrid: the rest of ns-2.29 mainline, but a 2001-vintage `diffserv/`
directory.  See `provenance/FW_THESIS_REFERENCES.md` §"Fork-divergence
evidence" for the full provenance trace.

The three penalty modes in `SFDPolicy::applyPolicer()` / `FWPolicy::applyPolicer()`:

1. **`downgrade2 == 0` (deterministic):** every excess packet → `downgrade1` DSCP
2. **`downgrade2 == 1` (probabilistic):** P(GREEN) = `CIR / bytes_sent`
3. **`downgrade2 == 2` (periodic):** counter 0→4 → downgrade, at 5 → reset + GREEN

All three modes were designed by Nortel Networks engineers (Peter Pieda
et al.) and shipped in stock ns-2.29.  The thesis (Chapter 3.3.2) does
not mention FW/SFD — see `provenance/FW_THESIS_REFERENCES.md` for the
full analysis.

## Improvements in the ns-2.35 port

The ns-2.35 port is framed as "improved DS4" rather than a faithful
reproduction of the 2001 defects. All improvements are in layer 2 at
`src/ns-2.35/`, which replaces selected DS4 base files
when the patch script is applied to the ns-2.35 tree. This makes the
ns-2.35 port the authoritative reference implementation against which
the ns-3 port is compared.

See `docs/HISTORICAL_BUGS.md` for full analysis of each defect.

The ns-2.35 port now covers all 14 files that DS4 modifies in ns-2 (the

### BUG-1: cbr_traffic.cc — set_apptype(PT_CBR)

The DS4 original has `set_pkttype(PT_CBR)` immediately overwritten by
`set_pkttype(PT_UDP)`. The ns-2.35 replacement changes the second call
to `set_apptype(PT_CBR)`, so CBR traffic is stamped with the correct
application type for DiffServ classification while keeping transport
type PT_UDP.

### BUG-2: realaudio.cc — set_apptype(PT_REALAUDIO)

Same pattern as BUG-1. The ns-2.35 replacement changes the second call
to `set_apptype(PT_REALAUDIO)`. Note: `PT_REALAUDIO=50` in ns-2.35
(was 49 in ns-2.29); resolved symbolically at compile time.

### BUG-3: ns-source.tcl — symbolic PT_FTP constant

The DS4 original uses the literal `27` for `set_apptype`. Verification
against `ns-2.35/common/packet.h` confirms `PT_FTP=27`, `PT_HTTP=31` —
so the value was semantically correct (earlier documentation incorrectly
described it as PT_HTTP). The ns-2.35 fix replaces the literal with a
Tcl variable `set PT_FTP 27 ;# symbolic constant mirroring common/packet.h`
and uses `$::PT_FTP` throughout, for readability and robustness.

### BUG-4: dsscheduler.cc — SFQ empty() before front()

In `dsSFQ::DequeEvent()`, the original calls `FlowQueue.front()` before
`empty()`, which is undefined behaviour in C++ when the queue is empty.
The ns-2.35 replacement reorders the check: `!empty()` is tested first,
and `front()` is called only inside the guarded branch.

### BUG-5: ns-source.tcl — complete FTP set_apptype

The DS4 original only patches `Application/FTP::start` with `set_apptype`.
The ns-2.35 replacement (combined with BUG-3) adds `set_apptype $::PT_FTP`
to all four instprocs: `start`, `send`, `produce`, `producemore`.

### UDP header overhead (+28 bytes)

`UdpAgent::sendmsg()` in the DS4 original sets `hdr_cmn::size()` to the
application payload only — no IP (20B) or UDP (8B) header bytes. ns-3
includes headers, making cross-simulator comparison inconsistent.

The ns-2.35 replacement adds `IP_HDR_SIZE + UDP_HDR_SIZE` (28 bytes) to
`hdr_cmn::size()` in both code paths of `sendmsg()` — both the full-packet
loop and the remainder fragment path. This makes UDP serialisation time
in ns-2.35 reflect real-world link usage and aligns with ns-3 behaviour.
